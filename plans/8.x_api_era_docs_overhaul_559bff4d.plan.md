# 8.x API Era Docs Overhaul

## What is wrong today (root causes)

1. **Era/stage label drift**: `analytics-platform.md` header says "Era 9 — Stages 7.1–7.9". It is an `8.x` doc covering stage `8.1`.
2. **Repetitive generic content**: `8.0 — API Era Foundation.md` – `8.9 — API Release Candidate.md` all share the same template text verbatim across five task tracks × eight services. The runtime flowcharts, ownership tables, and backend scope sections do not match each minor's actual roadmap summary.
3. **Wrong scope sections**: Every minor's "Backend API and Endpoint Scope" hard-codes `logsapi_endpoint_era_matrix.json` and "Frontend UX Surface Scope" hard-codes `logsapi-ui-bindings.md`, regardless of what the minor actually covers.
4. **Broken placeholders**: `emailapis-public-private-apis-task-pack.md` contains `$(System.Collections.Hashtable.era)` template strings.
5. **Truncated service set**: Version docs and task tracks only mention 8 services; the full Contact360 service set for `8.x` includes 14 codebases (`appointment360`, `connectra`, `jobs`, `mailvetter`, `contact.ai`, `emailapis`/`emailapigo`, `s3storage`, `salesnavigator`, `emailcampaign`, `logs.api`, `app`, `admin`, `root`, `extension`, `email`).
6. **Missing patch-level checklists**: No `8.n.0`–`8.n.9` granularity exists; the "Master Task Checklist" at the bottom of each version file is generic and identical across all minors.
7. **No codenames**: Versions have no short, memorable identifiers beyond "v8.N".
8. **Minimal task packs**: `connectra`, `jobs`, `logsapi`, `s3storage` task packs are stubs with 4–5 bullet points; the codebase analyses surface 5–7 concrete 8.x gaps per service.
